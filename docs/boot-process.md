# Linux Boot Flow

AmazonSpiceOx currently uses the direct QEMU kernel boot path plus a
persistent virtio root disk:

1. QEMU loads `out/bzImage` with `-kernel`.
2. QEMU passes `out/rootfs.cpio.gz` with `-initrd`.
3. QEMU attaches `out/rootfs.ext4` as a virtio block device.
4. The kernel decompresses itself and initializes early architecture code.
5. The kernel unpacks the initramfs into a temporary root filesystem in RAM.
6. The kernel executes `/init`.
7. `/init` mounts `/dev/vda` at `/newroot`.
8. `/init` runs `switch_root /newroot /sbin/init`.
9. `/sbin/init` becomes the real PID 1 and starts AmazonSpiceOx userspace.

The default command is generated by `scripts/run-qemu.sh`:

```sh
qemu-system-x86_64 \
  -m 512M \
  -kernel out/bzImage \
  -initrd out/rootfs.cpio.gz \
  -append "console=ttyS0 earlyprintk=serial,ttyS0,115200 panic=-1 init=/init root=/dev/vda rootfstype=ext4 rw" \
  -display none \
  -serial mon:stdio \
  -no-reboot \
  -netdev user,id=net0 \
  -device virtio-net-pci,netdev=net0 \
  -drive file=out/rootfs.ext4,if=virtio,format=raw
```

`console=ttyS0` sends kernel and userspace output to the serial console.

`earlyprintk=serial,ttyS0,115200` helps debug very early kernel output.

`panic=-1` prevents an immediate reboot after a kernel panic, which makes the
failure visible.

`init=/init` tells the kernel exactly which program should become PID 1.

`root=/dev/vda rootfstype=ext4 rw` is read by AmazonSpiceOx stage 1. The
kernel does not mount this root directly because we still boot through an
initramfs; stage 1 mounts it and hands off with `switch_root`.
